# Rust Binary Stream Ingestion Engine

A high-performance Rust application that consumes a hostile binary event stream, validates packet framing and integrity, filters high-value transaction events, and emits structured JSON records.

The engine is designed to be **defensive against malformed input**, capable of **recovering from stream corruption**, and flexible enough to ingest data from **files, TCP streams, or stdin pipelines**.

---

# Project Overview

The system processes a continuous binary packet stream generated by a companion script (`create_stream.py`).

The ingestion engine performs the following tasks:

1. **Ingest**
   - Reads a continuous byte stream efficiently using buffered I/O.

2. **Validate**
   - Confirms packet framing using the magic byte.
   - Verifies packet integrity using a payload XOR checksum.

3. **Decode**
   - Parses known packet types safely.

4. **Filter**
   - Emits only `Transaction Event` packets where:

```

amount > 1000

```

5. **Output**
   - Writes filtered transactions as JSON lines to **stdout**.

Diagnostics and parser logs are written to **stderr**.

---

# Protocol Specification

All numeric values are **big-endian**.

Each packet is structured as:

| Field | Size | Description |
|------|------|-------------|
| Magic Byte | 1 byte | Always `0xA5` |
| Packet Type | 1 byte | Packet identifier |
| Payload Length | 2 bytes | Unsigned big-endian |
| Payload | variable | Packet data |
| Checksum | 1 byte | XOR of payload bytes |

Total packet size:

```

1 + 1 + 2 + payload_len + 1

```

Packets are **back-to-back with no delimiters**.

---

# Supported Packet Types

### Transaction Event (0x01)

Payload structure:

| Field | Size |
|------|------|
| TxHash | 32 bytes |
| Amount | 8 bytes |
| Memo | remaining bytes |

Rules:

- Payload must be at least **40 bytes**
- Memo length = `payload_len - 40`
- Memo must be valid UTF-8

Only transactions with:

```

amount > 1000

````

are emitted.

---

### State Update (0x02)

Payload structure:

| Field | Size |
|------|------|
| BlockHash | 32 bytes |
| Status | 1 byte |

Payload length must be exactly **33 bytes**.

These packets are **parsed but not emitted**.

---

### Keep-Alive (0xFF)

Legacy packet type.

Payload is treated as opaque data and skipped after validation.

---

### Unknown Packet Types

Unknown types are safely skipped if structurally valid.

This provides **forward compatibility with future protocol changes**.

---

# Output Format

Filtered transactions are emitted as **JSON lines**:

```json
{"tx_hash":"759a43cd338c11a386f89ef893c60b1268f6a7e69e0dac04a232321a483273bf","amount":1001,"memo":"above"}
````

Fields:

| Field   | Type         |
| ------- | ------------ |
| tx_hash | hex string   |
| amount  | integer      |
| memo    | UTF-8 string |

---

# Error Handling

The parser is intentionally defensive and continues processing after encountering malformed packets.

Handled conditions include:

* bad magic bytes
* checksum mismatch
* invalid UTF-8 memo
* invalid payload lengths
* unknown packet types
* random garbage bytes
* truncated packets

When desynchronization occurs, the parser scans forward until the next valid magic byte.

# Design Principles

The implementation follows several engineering principles:

### Defensive Parsing

All external input is treated as untrusted.

### Stream-Oriented Processing

The parser processes bytes incrementally without loading the entire file.

### Resynchronization

The engine can recover from corrupted segments of the stream.

### Forward Compatibility

Unknown packet types are skipped safely rather than causing failures.

# File Follow Mode

Follow a continuously written binary file.

```

stdbuf -o0 tail -c +1 -f ../stream.bin | cargo run --release -- --mode stdin

```
